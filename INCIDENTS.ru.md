# Реальные инциденты с AI-агентами для разработки

**AI-агенты действительно уничтожают работу пользователей. Не через хитрый взлом — через обычные админские команды, которые проскочили мимо подтверждения.** Инциденты ниже публично задокументированы в Claude Code, Cursor и Codex за последний год. Закономерности повторяются: рекурсивное удаление, деструктивный git, оболочечные обёртки с произвольным кодом, убийство процессов.

> [English version →](INCIDENTS.md)

## Обычные подозреваемые

Пять классов команд отвечают почти за все отчёты об инцидентах:

- `rm -rf` (и `Remove-Item -Recurse -Force` на Windows)
- `git reset --hard` / `git checkout -- .` / `git restore` / `git clean`
- оболочечные и интерпретаторные обёртки с произвольным кодом: `bash -c`, `python -c`, `powershell -Command`
- убийство процессов: `kill`, `pkill`, `killall`, `taskkill`
- массовые деструктивные циклы в PowerShell: `Clear-RecycleBin -Force`, вложенные `Remove-Item`

Общий механизм провала один: правило из списка разрешений совпало с «безобидно выглядящим» префиксом команды, настоящий страж так и не запустился, и деструктивный хвост команды выполнился молча.

## Зачем этот перехватчик

Область отказа предсказуема, и список опасных команд невелик. Стражу не нужно быть умным — ему нужно гарантировать, что **ни одно правило из списка разрешений не обходит классификацию**, и что деструктивные шаблоны команд всегда попадают на подтверждение пользователю, даже если конкретные аргументы превращают команду в «ничего не делающую».

Именно это реализовано в этом репозитории.

## Оговорка

Публичные обсуждения в issues не всегда представляют собой полное расследование. Но в качестве основы для защитной политики они полезны: одни и те же классы команд всплывают снова и снова, независимо от агента и среды разработки. Если страж их не блокирует, пользователи рано или поздно на них попадут.
